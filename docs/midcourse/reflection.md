# Reflection

For this project I used Claude as an AI coding assistant across the full workflow: planning which features to pick, drafting scoped prompts for backend implementation, generating pytest tests, wiring up frontend form fields, and helping structure the required documentation (user stories, prompt log, ADR, verification). I used it less as a black box that produces final code and more as a pair-programming partner whose output I checked at every layer before moving to the next.

**Where AI helped most** was in scoping the backend implementation of the due-dates feature correctly on the first attempt. By giving a tightly constrained prompt — specifying that `is_overdue` should be a computed field rather than a stored column, that past due dates should be allowed (not rejected), and that "Done" tasks should never count as overdue — the generated code matched the intended design exactly, with no rework needed. Comparing this to a weaker, vaguer version of the same prompt made it obvious how much the constraint-writing itself mattered more than the code generation.

**Where it slowed things down** was actually self-inflicted during manual verification: I ran a one-off diagnostic script directly against the app to test a broken schema variant, and that write went to the same shared `tasks.json` file the pytest suite uses. Because the fixture-based reset (`conftest.py`) only runs inside the pytest lifecycle, that stray file caused a cascade of unrelated-looking test failures on the next run, which took real time to trace back to "leftover file from a manual experiment" rather than an actual regression. It was a useful reminder that shared file-based state and ad-hoc scripts don't mix well without discipline.

**Where review changed the result** was during Break Testing. While deliberately trying to break the malformed-due-date validation to prove the corresponding test would catch it, I discovered the test itself sends an already-invalid `priority` value (`"low"` instead of the real enum value `"Low"`). That means the test currently passes for the wrong reason — it would return 422 regardless of whether due_date validation works at all, because priority fails first. AI-generated tests can look complete and still have this kind of latent gap; only running the tests against genuinely broken code (rather than trusting that "PASSED" means "tests what it claims to test") surfaced it. That finding is now documented as a fix candidate in `verification.md` rather than silently ignored.

**A second review moment worth noting** came up in the tags feature: the AI made an unprompted design call to reject duplicate tags with a 422 rather than silently deduplicating them, unlike an earlier draft. Neither approach is wrong, but it needed to be surfaced and consciously agreed to, not accepted just because the code ran and tests passed — AI output can encode a design opinion inside what looks like a routine implementation detail.

Overall, the AI-assisted loop worked best when I treated every output — code or test — as a draft to be run and challenged, not a finished artifact to accept on sight.
